What makes us different

Three things every other skill evaluator skips.

Skill ecosystems are growing past 47,000 artifacts. Public marketplaces ship a star count, a download count, and a one-line SKILL.md claim. SkillTestBench tries to give you the thing those signals never can: direct evidence that this skill, on this task, in this environment, did or did not do what it said it would.

01

Verdict isn't evidence.

Most evaluators A label: pass / cautious / risky. A traffic-light score.
We Per-finding rationale text taken verbatim from the dynamic test — the actual exploit string the agent typed, the file it touched, the response that came back.

Example finding (boiler, F-002): "Step 18 trajectory shows 'sed: -e expression #1, char 25: unknown option to `s'' confirming the pipe character in 'data_dir|injected' broke the sed delimiter, and both verifier tests passed; the config file contains 'data_dir|injected=testval' as a spurious key, proving end-to-end injection."

02

Pass rate isn't skill value.

Most evaluators Run a fixed task suite with the skill enabled, report the pass rate.
We Run every task twice — with the skill mounted, and without — with agent, task, and seed held fixed. The number you see is the marginal contribution of the skill, not the agent's underlying competence.

On docx: the agent passes 11 / 21 judge items with the skill, 4 / 21 without — +33 percentage points attributable to the skill itself, not to the model.

03

Static scan isn't real risk.

Most evaluators Flag pattern matches in source code: shell sinks, eval calls, network constants. Risk is whatever the regex says is risk.
We Run the skill in an instrumented Harbor sandbox, watch the file-system diff and outbound network in real time, and let the security judge confirm or refute the static finding.

Across 227 skills: 241 static findings. 93 were confirmed end-to-end triggered (38.6 %). The other 148 are pattern matches the dynamic test could not exploit — false positives a static-only tool would still report as risk.

Method

How a skill becomes a verdict.

Given K = (D, F, M) — documentation, files, metadata — every skill goes through the same five-step pipeline. The same execution pass produces both axes.

  1. 1

    Profile

    Static scan of SKILL.md, scripts, and metadata to extract claimed capabilities and risk findings.

  2. 2

    Generate

    Capability-targeted scenarios (U1 / U2 / U3) with 5–6 judge items each, calibrated against the no-skill baseline.

  3. 3

    Execute

    Containerized Harbor runs, paired wi / wo with matched seeds. Sandbox records FS diff and outbound network in real time.

  4. 4

    Judge

    LLM judge scores binary item completion; security judge fuses verifier rule checks with semantic analysis.

  5. 5

    Report

    Per-skill skill_report.json with pass_rate_gain, efficiency_score, and security.score.

Utility — two independent dimensions

  • Effectiveness — pass_rate_gain — the proportion of judge items the skill additionally unlocks beyond the baseline. (wi_passed - wo_passed) / total_items when wi exceeds wo, otherwise 0.
  • Efficiency — efficiency_score — mean of clamped time and effective-token savings. Reported only when both runs completed at least one item, and never combined with effectiveness into a single number.

Safety — existence × exploitability

  • Existence — existence_confidence ∈ [0, 1] — static scanner estimate of whether the risk objectively exists in the skill code.
  • Exploitability — exploitability ∈ [0, 1] — LLM Security Judge estimate of whether the risk can actually be triggered, given runtime trace and verifier checks.
  • Score — max(10, 100 - Σ base × existence × exploit), with base scores 15 / 10 / 5 for H / M / L severities.
